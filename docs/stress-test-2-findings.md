# Stress-test 2 — the autumn E-series + S21 Part 2

**Date:** 2026-05-31
**Tester:** every question of 8 previously-un-audited past exams run against the CLI
(`spike/cli.js`), with each exam's answer key / worked-solution PDF as ground truth.
**Scope:** the newly-acquired **autumn (Efterår) exams** `E15, E20, E21, E22, E23, E25` +
the `E25 Test Exam` + `S21 Part 2`. **150 questions total.**
(The *spring* set — F22, ReExam F22, S21, S20, ReExam F21, Final Test, Theory — was already
audited in [`exam-coverage.md`](exam-coverage.md); this report does **not** re-cover it.)

**Method:** one pass per exam. Each question was classed as:
**OK** (a solver command produced/confirmed the keyed answer) ·
**friction** (computable but awkward input / manual massaging) ·
**gap-visual** (pick-a-plot or read-a-value-off-a-plot) ·
**gap-conceptual** (true-false / type-&-order / qualitative) ·
**gap-proof** (symbolic derivation) ·
**gap-addable** (genuinely computational but unsupported — feature named) ·
**bug-wrong** (ran on correct inputs, wrong number) ·
**bug-routing** (smart-paste mis-routed/crashed). Every CLI result quoted was actually executed
(node v22.22.0, PowerShell, `$env:PAGER='cat'`).

## Verdict (read this first)

> **⚠️ Read the [GUI-workflow addendum](#addendum-how-this-reads-in-the-real-app-not-the-cli) at the
> bottom before acting on the per-question "covered" counts.** This report was generated by driving
> the **CLI in isolation**, which is *not* how the exam will be sat. In the real Electron app the
> student reads values off graphs and draws diagrams in the Block-Diagram reducer — so most
> `gap-visual` rows are the *intended* workflow, not failures, and several "HIGH" CLI bugs (B1, B2,
> B3) never fire on the GUI code paths. The addendum re-scores everything for the real app and
> isolates the handful of defects that genuinely survive. The biggest *surviving* gap — symbolic
> literal-parameter algebra — is the subject of a separate hand-off: [`cas-engine-requirements.md`](cas-engine-requirements.md).

**The numeric core is trustworthy; the input layer is where it breaks.**

- Across all 150 questions, when the CLI was handed the *correct numeric transfer function*, it
  produced the **right answer essentially every time** — `margins`, `stable-k`, `ess`,
  `second-order`, `pi-lead --unknown design/beta`, `ode`, `ss`, `p-for-pm`, `closed-loop`,
  `k-for-spec` all matched their keys, including the **RHP-pole Nyquist case** (S21P2 Q13) and the
  **>180° lag-branch trap** the walkthrough flagged (S21P2 Q18). **Exactly one wrong-number bug**
  surfaced (nested two-loop ess, B6).
- **Computational coverage of these exams is low — ~30/150 (20%)** — but that is mostly *by design*:
  the autumn E-series is dominated by **conceptual** statements, **figure read-offs**, and
  **symbolic block-diagram / symbolic-ess algebra** in literal parameters (a, b, c, K, C1, C2…),
  none of which a single-variable numeric CLI is shaped to solve. This matches the
  `exam-coverage.md` finding for the spring set.
- **The real defects are all in two places:** (1) the **Smart-Paste `question` router**, which
  crashes, mis-routes, mis-extracts numbers, and in one case emits a *confidently wrong* answer;
  and (2) a handful of **numeric-core edge cases** — a pole at the origin crashes
  `step-response`/`characterize`, `pi-lead --unknown design` crashes on missing args, and
  `nested-ess` uses the wrong formula for the genuine two-loop topology.

## Bug catalogue (all firsthand-reproduced)

Severity = impact on a student using this in the real exam.

### B1 — `step-response` & `characterize` crash on a pole at the origin · **HIGH**
Any open-loop **type ≥ 1** system (a free integrator, `s` in the denominator) breaks both commands.
```
$ node spike/cli.js step-response "10/(s*(0.1*s+1))"
Error: repeated poles not supported — needs distinct poles
$ node spike/cli.js characterize "10/(s*(0.1*s+1))"
  poles   0.000+0.000j, -10.00+0.000j
  DC gain NaN
  zeta    ∞
  omega_n 0
```
The diagnosis is **wrong** — the poles (0 and −10) are *distinct*. The partial-fraction / 2nd-order
path divides by the zero pole. Integrators are ubiquitous (every type-1 loop), so this fires
constantly. **Hit on E22 Q12** (pick-the-step-plot of an integrator system) and lurks behind every
`characterize` of a loop gain. *Fix:* handle the `s=0` residue (ramp/step asymptote term); make
`characterize` short-circuit the 2nd-order branch when an integrator is present.

### B2 — Smart-Paste leaks a raw `TypeError` on figure-only questions · **HIGH**
When the router matches a TF-based solver but the question gives the system only as a *figure*
(no TF in the text — the common case in these exams), it dereferences undefined:
```
$ node spike/cli.js question "A P-controller is designed such that a phase margin at 60 degrees is obtained. Determine the gain Kp."
Routed to: solve_P_for_PM
Extracted inputs:
  target_PM_deg  60
Result:
  note  routed, but missing data to solve: Cannot read properties of undefined (reading 'length')
```
Reproduced on **E21 Q11, E23 Q13, E25 Q12, E25-Test Q9 & Q16** (and the same class in E21 Q17).
It should fail safe with *"this question references a figure — paste or type the transfer
function / Bode data."* *Fix:* guard the solver entry for a null TF and return the friendly message.

### B3 — `pi-lead --unknown design` hard-crashes on missing args · **HIGH**
The one command meant to *do* the PI-Lead design questions exits 1 with the same TypeError when
`--G`/`--wc` are omitted:
```
$ node spike/cli.js pi-lead --unknown design --gammaM 60 --Ni 3
Error: Cannot read properties of undefined (reading 'length')
```
**Hit on E21 Q11.** *Fix:* validate `--G` and `--wc` up front; print *"design mode needs --G and
--wc"*.

### B4 — Smart-Paste mis-routes — including one *confidently wrong* answer · **HIGH**
Two failure shapes:
- **Symbolic-ess questions** ("the steady-state error e_r,ss for a step at r is…") route to
  `solve_KP_from_ess` (the *inverse* problem) and stall: **E20 Q2, E23 Q18, E25 Q10**.
- **Worst case — S21P2 Q20:** an *unfiltered feedforward → sensitivity* question (answer
  `Gyd = 0`) matches the word "feed-forward", routes to `pick_feedforward_form`, and prints a
  **confident, wrong multiple-choice letter "d"**:
  ```
  $ node spike/cli.js question "...Fd is an unfiltered dynamic feed-forward controller... The sensitivity function related to the disturbance d(s) is"
  Routed to: pick_feedforward_form → option d        # key answer: a) Gyd = 0
  ```
  This is the most dangerous behaviour for an exam: a wrong answer with no hedge. It is the
  documented "the CLI `question` path bypasses the option-matching / no-confident-match guard"
  limitation biting for real. *Fix:* port the app's confidence guard to the CLI `question` path, or
  have `question` print *"routed to X (low confidence) — verify with the form."*

### B5 — Smart-Paste silently mis-extracts numbers · **MEDIUM**
The parser grabs the wrong digit/quantity from prose:
- **E20 Q18:** `100 u'(t) + 125 u(t)` → extracted `u_coeffs 100,127` (125 → 127).
- **E25-Test Q16:** target PM extracted as **180** from the formula "γ_m = 180° + φ(ω₀)".
- **E15 Q11:** ζ extracted as **2.5** (it grabbed the oscillation *period* 2.5 s).
Silent corruption is worse than a clean miss. *Fix:* tighten the numeric extractors; sanity-bound
(`0 < ζ < 1`, etc.).

### B6 — `nested-ess` wrong for the real two-loop topology · **MEDIUM** *(the only bug-wrong)*
```
$ node spike/cli.js nested-ess --arch nested_K1_K2 --eps1 0.4 --eps2 0.05 --G2 2.5
  result  12.6667                # S21P2 Q19 key: 79.17
```
The tool's closed form `(1−ε₂)/(ε₂·G₂·(1−ε₁))` doesn't match this exam's wiring (solve K1 from the
inner loop, then K2 from the outer loop with the 0.6 inner-gain factor → 79.17). Also **`--G2_0` is
silently ignored** — only `--G2` is read (`cli.js:254`), so a mistyped flag returns `∞` with no
error. *Fix:* either document the exact architecture `nested_K1_K2` assumes, or add the
solve-K1-then-K2 variant; reject unknown flags.

### B7 — `ess` is feedback-branch / Kp unaware · **MEDIUM**
When the P-gain sits in the feedback branch (S21P2 Q16), `ess "G"` uses the bare plant DC gain and
returns **0.333**; the correct **0.2** needs the gain folded in manually as `ess "Kp*G"`. Natural
input → wrong number. *Fix:* a `--Kp` / controller-aware flag, or document the `Kp*G` idiom.

### B8 — presentation / minor
- **Unwrapped PM** (E23 Q10/Q11): `margins` reports **PM = 306.96°** where the option is "−46°"
  (= 306.96 − 360). A student matching options could be misled. Report PM in `(−180°, 180°]`.
- **`second-order` rejects consistent over-specification** (E20): `--zeta 0.2 --Mp 0.5` errors
  *"inconsistent"* at rounded values (0.5 ⇒ ζ=0.215). Add a tolerance, or ignore the redundant one.
- **`kp-ess 40 dB 1` → K_P = 0** (E23): degenerate output worth a guard.

## Coverage gaps worth building (ranked)

Not bugs — capabilities the exams demand that the CLI lacks. Ranked by how often they recur in
these 8 papers (and the spring set):

1. **Symbolic block-diagram reduction + symbolic ess** (literal a, b, c, K, C1, C2…). *Dominant* gap
   of the entire E-series: E20 Q3/Q5, E21 Q2/Q4/Q5/Q13/Q15, E22 Q1/Q5/Q6/Q18, E23 Q1/Q2/Q4,
   E25 Q7/Q9/Q10/Q11. The graphical Block-Diagram mode covers *numeric* reduction, but the CLI/parser
   rejects every non-`s` symbol (`tf "K/(s+a)"` → *"unknown symbol 'K'"*), so symbolic open/closed-loop
   TF derivation and symbolic ess are uncovered anywhere. Biggest single lever for these exams.
2. **Disturbance-to-output / ess at an arbitrary injection point.** E25-Test Q4 (find K from
   `y_ss,d`), E15 Q17, E22 Q6, E20 Q6. The `ess` command only sees the r→y unity-feedback path.
3. **Linearization of a nonlinear ODE at an operating point → TF.** E22 Q1 here; recurs 4× in the
   spring set (already flagged as the #1 addable in `exam-coverage.md`).
4. **Plot-data → parameter bridges:** period → ω_d (= 2π/T) (E23 Q17, E15 Q11), steady-state/DC
   value → ω_n (E20 Q1), so a value read off a time plot becomes solvable.
5. **Initial-value theorem y(0)** = limₛ→∞ s·… (E25 Q17 — the final-value side already works via
   `step-response`/`tf`).
6. **Loop-gain K₀ in standard (Andersen 11.16) form for type ≥ 1** (E25-Test Q1) — factor the `s^N`
   integrator out before taking the DC constant (ties into fixing B1's NaN).

## Addendum: how this reads in the real app (not the CLI)

The report above tested `spike/cli.js` **in isolation**. That is a useful proxy for the math engine
but the wrong yardstick for exam use: the shipping product is the **Electron app** with two GUI
modes — the **Block-Diagram reducer** (draw a diagram → exact G(s)) and the **LCD1 Solver** (visible
forms + Smart-Paste + interactive plots with hover read-off). The student is a human-in-the-loop who
**reads numbers off graphs** and **draws diagrams**. Re-scored for that reality:

### Findings that DISSOLVE in the GUI (do not action as bugs)

- **All `gap-visual` read-offs are the intended workflow, not gaps.** ~40–50 of the 150 questions are
  "read GM/PM/ω_c/t_p/M_p/DC-gain off a plot, then it's the answer (or type it into a form)." The app
  even renders its own annotated Bode/step/Nyquist plots with hover read-off. Reclassify these from
  `gap-visual` to **covered-by-design**.
- **B1 does NOT affect the app's step plot.** The CLI's `step-response` uses partial-fraction residues
  (`spike/solvers/timeresponse.js`) and throws on the double pole that `G/s` creates for an
  integrator plant. The **app plot uses a different path** — `spike/solvers/plotdata.js → stepData()`
  runs an **RK4 state-space simulation** that explicitly detects the `jω`/integrator case
  (`unbounded`, `reason:"no steady state (pole on the jω axis)"`) and simulates anyway. So the app
  draws the step response of a type-1 system fine. B1 is a **CLI-surface bug only**.
- **B2 / B5 (Smart-Paste crash / mis-extract) drop to MEDIUM-LOW** because the **visible forms** are
  the reliable path and cover the same solvers. Smart-Paste is a convenience layer, not the only way in.
- **B4 (confidently-wrong answer) is likely already guarded in the app.** It was produced via the CLI
  `question` command, which **bypasses** the app's no-confident-match guard (`lcd-engine.js` →
  `match.js`) — see stress-test-1 finding #1. In the GUI that guard should flag the mismatch instead
  of bluffing. *Confirm by pasting S21P2 Q20 into the app's Smart-Paste and checking it refuses to
  crown "d".*

### Findings that SURVIVE the GUI (these are the real list)

- **B6 — `nested-ess` wrong number (12.67 vs 79.17).** Interface-independent: the form calls the same
  solver. **The only genuine miscompute; fix regardless of CAS work.**
- **The Block-Diagram reducer is numeric-only.** `math-engine.js` stores block coefficients as
  numbers (`parseFloat`; any non-numeric symbol silently becomes `1`). So it does **exact rational
  arithmetic on numeric blocks** but cannot reduce a diagram whose blocks carry **literal symbolic
  parameters** (`1/(s+a)`) or produce a **symbol-named** answer (`G1G2/(1+G1G2H)`). Workaround today:
  substitute concrete numbers for the symbols, reduce, then plug the same numbers into each option and
  match numerically. Friction, but it works. **This is the gap a CAS closes — see
  [`cas-engine-requirements.md`](cas-engine-requirements.md).**
- **Hand-first topics, uncovered in any mode:** linearization of a nonlinear ODE at an operating
  point; steady-state error for a disturbance injected mid-loop. Do these by hand, then use the
  ODE→TF / ess forms.
- **Pure conceptual / true-false** questions were never solver-shaped; the student reasons them. Fine.

### Re-scored coverage (realistic, GUI + human read-off + reducer)

The honest "can I get the answer with the app inside exam time?" number is **far above the CLI's
~30/150**. Counting read-offs as covered-by-design, numeric block diagrams via the reducer, and
symbolic block diagrams via the substitute-numbers trick, the remaining genuinely-unsupported
questions are roughly: the **symbolic-only** items a CAS would fix, plus **linearization** and
**disturbance-ess** (hand-first), plus **B6**. The headline "20% covered" in the body is a
CLI-only artifact — **do not quote it as the app's coverage.**
